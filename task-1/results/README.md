### **ADR: Выбор Apache Airflow для оркестрации пайплайнов данных**

---

## **Контекст**
Компании требуется платформа оркестрации для управления ETL/ELT-пайплайнами, которая должна:
- Предоставлять возможность гибкой настройки пайплайна;
- Интегрироваться с **BigQuery**, **Redshift**, **Kafka** и **Spark**;
- Поддерживать встроенный мониторинг и оповещения;
- Эффективно обрабатывать до **1 млн записей** за один запуск пайплайна.

---

## **Альтернативы**

| Технология | Описание | Плюсы | Минусы |
|:-|:-:|:-:|:-:|
| **Spring Batch** | Фреймворк на Java для пакетной обработки данных. | – Глубокая интеграция с Java-экосистемой, Spring Framework<br>– Строгая типизация, тестируемость<br>– Встроенные механизмы повторов и skip-логики | – Требует написания кода на Java/Kotlin, что увеличивает порог входа для дата-инженеров, привыкших к Python<br>– Нет встроенного веб‑интерфейса для мониторинга и управления пайплайнами<br>– Сложнее организовать ветвление, event‑триггеры и гибкую оркестрацию DAG‑зависимостей<br>– Интеграции с BigQuery, Redshift, Kafka, Spark требуют кастомной реализации или использования сторонних библиотек |
| **Kubernetes Jobs (CronJob)** | Нативные ресурсы Kubernetes для запуска одноразовых задач и периодических заданий. | – Простота развертывания в облаке, полная изоляция задач в контейнерах<br>– Встроенные механизмы повторов, возможность горизонтального масштабирования<br>– Не требует отдельной платформы оркестрации | – Отсутствует встроенный веб‑интерфейс для мониторинга DAG, логов, ручного управления<br>– Сложно реализовать зависимости между DAG, ветвление, условные операторы<br>– Нет встроенной поддержки email‑уведомлений, fallback‑логики<br>– Нет готовых модулей для работы с BigQuery, Redshift, Kafka, Spark |
| **Apache Spark** | Распределённый вычислительный движок, который также может выступать как оркестратор. | – Мощная обработка данных "in‑memory", идеально подходит для трансформаций большого количества записей<br>– Встроенные интеграции с BigQuery, Redshift, Kafka, а также Spark SQL | – Не предназначен для оркестрации разнородных пайплайнов: слабая поддержка ветвления, условных операторов, триггеров по событиям<br>– Отсутствует удобный UI для мониторинга DAG-зависимостей на уровне задач<br>– Нет встроенных механизмов fallback‑логики и email‑уведомлений<br>– Требует выделенного кластера. |
| **Apache Airflow** | Платформа оркестрации DAG на Python с богатой экосистемой провайдеров, веб‑интерфейсом и встроенными механизмами надёжности. | – Полностью соответствует требованиям<br>– Готовые провайдеры для BigQuery, Redshift, Kafka, Spark<br>– Гибкое описание пайплайнов на Python, мощный веб‑UI<br>– Облачные управляемые версии снижают эксплуатационную нагрузку | – Требует отдельной инфраструктуры (метабаза, брокер сообщений, воркеры)<br>– При очень большом количестве DAG планировщик может стать узким местом<br>– Не предназначен для низколатентной потоковой обработки |

## **Решение**

Airflow выбран благодаря балансу широкой экосистемы, поддержкой облачных сред и необходимым набором функциональных возможностей.

Apache Airflow обладает следующими преимуществами перед рассмотренными альтернативами:
1. **Интаграция с внешними API** – провайдеры (Apache Airflow providers) содержат готовые операторы для **BigQuery**, **Redshift**, **Kafka**, **Spark**. Это значительно ускоряет разработку и снижает количество кастомного кода.
2. **Гибкость в описании пайплайнов** – DAG пишутся на Python, что позволяет реализовать любую логику: ветвление (BranchPythonOperator), условные операторы (ShortCircuitOperator), динамическое создание задач, event‑триггеры через сенсоры и триггеры (deferrable operators).
3. **Надёжность** – встроенная поддержка повторов (retries), fallback‑логики (on_failure_callback), email‑уведомлений (email_on_failure, email_on_retry, кастомные callback).
4. **Масштабируемость** – при использовании CeleryExecutor или KubernetesExecutor Airflow может обрабатывать тысячи задач параллельно, что достаточно для наших объёмов.
5. **Облачная экосистема** – Airflow успешно развёртывается в любом облаке: управляемые сервисы (Google Cloud Composer, Amazon MWAA) или самостоятельная установка на Kubernetes с использованием официального Helm‑чарта.
6. **Сообщество и зрелость** – Airflow является de‑facto стандартом в области оркестрации пайплайнов, имеет большое комьюнити, обширную документацию и множество примеров.

### **Развёртывание в облачной среде**

Выбран гибридный подход с использованием **Kubernetes** (KubernetesExecutor) для максимальной гибкости и автономности:
1. **Контейнеризация** – все компоненты Airflow (web, scheduler, воркеры) упаковываются в Docker-образы на основе официального образа Apache Airflow.
2. **Оркестрация в Kubernetes** – Helm-чарт Airflow устанавливается в кластер Kubernetes (Google Kubernetes Engine, Amazon EKS или другие).
   - **Scheduler** и **Webserver** работают как стабильные поды.  
   - **KubernetesExecutor** для каждой задачи создаёт отдельный под, который автоматически удаляется после выполнения. Это обеспечивает изоляцию ресурсов и масштабирование под нагрузку.
3. **Метабаза** – управляемая PostgreSQL (Cloud SQL / RDS) для высокой доступности.
4. **Брокер сообщений** – Redis или RabbitMQ в кластере или как управляемый сервис (например, Memorystore, Amazon MQ).
5. **Мониторинг** – отслеживание DAG-ов и их состояние через AirFlow Web UI, встроенные метрики Airflow экспортируются в Prometheus (через StatsD).

**Объём данных** – обработка до 1 млн записей за запуск. Airflow оркестрирует задачи, но не обрабатывает данные напрямую. Нагрузка ложится на Spark (распределённая обработка) и облачные хранилища/БД (BigQuery, Redshift). Airflow с KubernetesExecutor способен управлять тысячами подов, что позволяет распараллелить обработку при необходимости.

## **Недостатки, ограничения, риски**

1. **Сложность настройки и эксплуатации** – требуется управление метабазой, брокером сообщений, настройка секретов, масштабирование воркеров.
2. **Производительность планировщика** – при очень большом количестве DAG (тысячи) планировщик может становиться узким местом.
3. **Не предназначен для низко-латентной потоковой обработки** – Airflow ориентирован на пакетные процессы. Хотя есть сенсоры, постоянный опрос событий может создавать излишнюю нагрузку. Для потоковых сценариев потребуются отдельные инструменты (Kafka Streams, Flink).
4. **Риск потери выполнения при сбое** – если сбой происходит до того, как состояние записано в метабазу (редко), может потребоваться ручное восстановление.
5. **Управление зависимостями** – разные DAG могут требовать разных версий библиотек Python. Использование KubernetesExecutor с изолированными контейнерами или виртуальными окружениями (venv) требует дополнительной конфигурации.
6. **Стоимость облачной инфраструктуры** – при активном использовании KubernetesExecutor может создаваться много короткоживущих подов, что увеличивает затраты на оркестрацию.
