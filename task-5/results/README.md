### **Выбор метрик**
Для мониторинга сервиса пакетной обработки данных отлично подойдет метод RED, так как в этом методе отслеживаются самые важные показатели:
- **Частота запросов** - позволит следить за периодическими изменениями нагрузки и узнавать о пиковых значениях, будет служить индикатором необходимости горизонтального масштабирования
- **Ошибки** - количество ошибок важный параметр, способный указать на проблемы в моделях данных или интеграции сервисов
- **Длительность** - необходимо для контроля соблюдения временных рамок установленных для пайплайна пакетной обработки

Логировать необхдимо события и их контекст - что произошло, какие данные сопутствовали событию и где это произошло. Такой информационный набор позволит изучить причины сбоев в системе и путь их возникновения.

### **Выбор способа отправки логов и конфигурации оповещений**
Чтобы обеспечить возможность исследования причин нештатной работы приложения или сбоев, логи должны содержать следующую информацию:
- FATAL — отказ сервиса или оборудования.
- ERROR — ошибочное состояние.
- WARN — состояние, которое близко к нестандартному поведению системы.

Следующие категории логов не используются для экономии места, но их можно добавить в тестовой среде до выпуска в продакшн для упрощения отладки:
- INFO — это штатное поведение. Такие логи обычно используют для записи обычных сообщений.
- DEBUG подробная информация о происходящем в программе
- TRACE обычно используют в среде разработки и для отладки в тестовом окружении.
